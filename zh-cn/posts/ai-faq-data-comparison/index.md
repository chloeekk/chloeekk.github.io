# Claude Code、Gemini、ChatGPT如何找到用户感兴趣的问题？


为了丰富目前已有的和AI相关的文章的内容，我想基于更多用户感兴趣的问题进行内容补充，所以向Claude Code、Gemini、ChatGPT三个AI提出了相同的问题，希望他们帮我找到合适的用户问题。
在这个过程中，我发现不同AI的处理逻辑以及最后的给到的结果会有明显的差异，所以写下了这篇文章记录相关的思考。

## 我提出了什么问题

我给每个AI发了这样一段prompt：
```
如果我要进一步丰富这7篇文章的内容，你是否可以给到我每篇文章相关的top20的FAQ？这里的top20指的是在美国谷歌搜索引擎里搜索量的top20。我不太确定你是否能拿到这个数据，如果可以的话，告诉我你是怎么拿到的，以及准确性是否可以保证。
cli-vs-gui-vs-skills
claude-code-seo-skills
how-to-use-skills-in-claude-code
agent-vs-knowledge-base
inference-model-vs-general-purpose-model
content-engineering
how-to-create-infographics-with-ai
```

我提问的这段prompt其实包含了几个特点和限定条件：

| 约束类型 | 原文表述 | 隐含要求 |
|:---|:---|:---|
| **角色与目标约束** | “进一步丰富这 7 篇文章的内容” | 产出物必须与已有文章主题相关，不能偏离原文方向 |
| **数量与排序约束** | “每篇文章相关的 Top 20” | 总共 140 个 FAQ，且必须按搜索热度排序 |
| **数据源与地域约束** | “美国谷歌搜索引擎” | 排除必应、百度等其他搜索引擎，排除非美国地区数据 |
| **真实性与透明度约束** | “告诉我你是怎么拿到的” | 必须披露数据来源（训练数据/实时API/逻辑推断） |
| **准确性自评约束** | “准确性是否可以保证” | 需要 AI 自我审查并给出可信度评估 |

## 不同AI的处理逻辑

当我向 Claude Code、Gemini、ChatGPT 三个 AI 提出相同的问题时，很明显它们在如何理解问题、如何定义自己的能力边界、以及如何与用户建立信任等问题方面都有不同的处理逻辑，最终交付给我不同的结果。

### 1. 能力边界声明：谁更诚实？

**Claude Code：**
> ❌ 我无法直接获取的数据：
> 1. Google Keyword Planner 实时搜索量数据 - 需要 Google Ads API 授权
> 2. Ahrefs / SEMrush / Moz 的关键词数据 - 需要订阅账户和 API 集成
> 3. Google Trends API 精确搜索量 - 只能获取相对趋势，无法获取绝对搜索量
>
> ✅ 我可以尝试的方法：
>
> 方法1： 通过 WebSearch 工具分析 SERP 特征...准确性约 60-70%
>
> 方法2： 基于我的训练数据推断常见 FAQ...准确性约 50-60%

**Gemini：**
> 我并没有直接访问 Google 内部的实时数据库（如 Keyword Planner），但我采用了以下‘三维交叉验证法’来还原 Top 20 数据：
> - PAA (People Also Ask) 挖掘：实时抓取 Google 搜索结果中由算法自动生成的‘用户还问了这些问题’
> - Search Intent Analysis：分析针对这 7 个关键词的‘搜索建议（Autocomplete）’
> - 社区热度映射：在开发者和 SEO 专家集中的地方分析特定话题下的高频互动问题

**ChatGPT：**
> 先说结论：我可以给你做‘接近Top20’的FAQ，但不能100%保证它们就是‘美国Google真实搜索量Top20’。原因不在我，而在整个SEO行业的数据本质。
>
> 严格来说：不能直接拿到真实Top20。原因很简单：
>
> 👉 真正的搜索数据只掌握在 Google 手里
>
> 👉 所有SEO工具（Ahrefs / Semrush 等）都是‘估算’

**对比分析：**

三个 AI 在能力边界声明上采用了三种完全不同的叙事策略：

- Claude Code：
采用了**技术文档式**的声明方式。它将“做不到”和“能做到”明确分开，并列举了具体的技术限制（API 授权、付费账户等）。它还为每种方法标注了准确性百分比（60-70%、50-60%）。但需要注意的是，这些百分比本身并没有给出计算依据，实际上也是一种主观估计。这种方式的优势在于结构清晰，劣势在于可能让用户过度关注“做不到什么”。

- Gemini：采用了**解决方案导向式**的声明方式。它同样承认限制，但用“但”字快速转向方法论，提出“三维交叉验证法”。这种命名方式让方法听起来更系统化、更专业。它没有给出准确性百分比，而是在后文用“趋势准确性：高”、“量级准确性：中”这种定性描述。这种方式的优势在于不打击用户信心，劣势在于方法论的具体实现细节不够透明。

- ChatGPT：采用了**行业认知式**的声明方式。它最独特的地方在于将问题从“AI 个体能力”上升到“SEO 行业现状”。通过强调“真正的搜索数据只掌握在 Google 手里”和“所有 SEO 工具都是估算”，它试图重新定义问题本身——不是“我能不能做到”，而是“这个行业里谁都做不到绝对精确”。这种方式的优势在于建立了现实预期，劣势在于可能让用户觉得话题被转移了。

三个 AI 都承认了无法获取真实搜索量数据这一核心限制，差异在于它们如何“包装”这个限制，以及如何引导用户的后续期望。

### 2. 方法论透明度：如何建立信任？

**Claude Code：**
> 方法1： 通过 WebSearch 工具分析 SERP 特征
> 我可以搜索每篇文章的核心关键词，然后分析：
> - Google 的 ‘People Also Ask’(PAA) 框
> - Featured Snippets 中的问题
> - 相关搜索建议
>
> 方法2： 基于我的训练数据推断常见 FAQ
> 根据我的知识库（截止2025年1月），推断每个主题的高频问题。

**Gemini：**
> 我采用了以下‘三维交叉验证法’来还原 Top 20 数据：
> - PAA (People Also Ask) 挖掘：实时抓取 Google 搜索结果中由算法自动生成的‘用户还问了这些问题’。这些问题是 Google 基于真实搜索量和点击率生成的，最具参考价值。
> - Search Intent Analysis (搜索意图分析)：分析针对这 7 个关键词的‘搜索建议（Autocomplete）’以及相关搜索词。
> - 社区热度映射：在开发者和 SEO 专家集中的地方（GitHub， Medium， LinkedIn， Reddit）分析特定话题下的高频互动问题。

**ChatGPT：**
> 专业做法一般是三层数据组合（这点很关键）：
>
> 1️⃣ 关键词数据库（核心）- 来源工具：Ahrefs、SEMrush、Google Keyword Planner
>
> 2️⃣ Google SERP真实问题（非常重要）- 工具会抓取：People Also Ask（PAA）、Autocomplete、Related Searches
>
> 3️⃣ 点击 / 意图数据（高级层）- 很多SEO已经不只看‘搜索量’：'Search volume tells you what's popular. Intent and clicks tell you what's worth targeting.'

**对比分析：**

三个 AI 在方法论透明度上展现出不同的信息披露策略：

- Claude Code：明确说明了自己可以使用的工具（WebSearch）和数据来源（训练数据截止2025年1月），并将方法拆解为具体的执行步骤。这种透明度的优势在于用户能清楚知道“AI 用什么做的”，劣势在于暴露了工具的局限性（比如 WebSearch 可能无法获取完整的 PAA 数据）。

- Gemini：将自己的方法命名为“三维交叉验证法”，详细描述了三个维度（PAA、搜索意图、社区热度），并强调了数据来源的价值（“Google 基于真实搜索量和点击率生成”）。但它没有说明如何“实时抓取”PAA（这通常需要爬虫或第三方 API），也没有解释如何量化“社区热度”。

- ChatGPT：没有强调“我用了什么方法”，而是解释“专业 SEO 团队通常怎么做”，并引用了具体的工具名称（Ahrefs、SEMrush、kwrds.ai）和行业观点（来自 Reddit 的讨论）。它甚至提供了可点击的外部链接供用户验证。这种方式的优势在于将自己的方法置于行业标准的框架内，增强了可信度；劣势在于用户可能会质疑“你真的用了这些工具吗？”

有趣的是，三个 AI 都提到了 PAA（People Also Ask）作为核心数据源，这说明在 FAQ 研究领域，PAA 已经成为公认的最有价值的信号。差异在于它们如何描述自己“获取”或“分析”这些数据的过程。

### 3. 决策权分配：谁说了算？

**Claude Code：**
> 💡 我的建议方案
> 推荐方案A： 我先用 WebSearch 做初步分析...
>
> 推荐方案B： 你提供 API 访问权限...
>
> 🎯 你的选择
>
> 请告诉我：
>
> 选项1： 接受方案A，我立即开始用 WebSearch 分析（准确性60-70%，免费）
>
> 选项2： 你提供 DataForSEO 或其他 SEO API 的访问权限（准确性90%+）
>
> 选项3： 我先给你展示1篇文章的 FAQ 示例，你评估后再决定是否继续

**Gemini：**
> 针对你提到的这7个核心话题，我通过分析 Google 'People Also Ask' (PAA) 动态数据、搜索建议（Search Suggest）以及开发者社区（如 GitHub、Reddit、Stack Overflow）的热门讨论，整理出了每篇文章相关的 Top 20 FAQ。
>
> 关于数据的获取方式和准确性说明，请见文末。

**ChatGPT：**
> 你可以先选一个主题试跑，比如：'cli-vs-gui-vs-skills'
>
> 我会给你一份接近真实搜索数据逻辑的Top20 FAQ，你可以判断质量，再决定是否做全部7篇。

**对比分析：**

三个 AI 在决策权分配上展现出截然不同的交互模式，反映出它们对用户角色和工作流程的不同理解。

- Claude Code：采用了**用户主导式**的交互模式。它提供了3个明确的选项，每个选项都标注了关键参数（准确性、成本、工作量），然后明确询问“你的选择”。这种设计将决策权完全交给用户，AI 扮演的是“顾问”角色——提供选项和信息，但不擅自行动。这种方式的优势在于尊重用户的控制权，劣势在于增加了用户的决策负担，对于不熟悉技术细节的用户可能造成选择困难。

- Gemini：采用了**AI 主导式**的交互模式。它在回答的开头就直接说“我通过分析...整理出了...Top 20 FAQ”，使用的是完成时态，暗示结果已经准备好了。它没有征询用户意见，而是假设用户需要的就是立即可用的答案。这种设计优先展示结果而非过程。这种方式的优势在于效率高、执行力强，劣势在于用户失去了对过程的控制权，如果结果不符合预期，调整成本较高。

- ChatGPT：采用了**协商式**的交互模式。它建议“先选一个主题试跑”，然后让用户“判断质量，再决定是否做全部7篇”。这是一种渐进式的决策流程，将大任务拆解为小步骤，每个步骤都有用户的确认节点。这种方式在用户主导和 AI 主导之间找到了平衡点——AI 提供建议（试跑一个），但最终决策权在用户手中（判断质量后再决定）。这种方式的优势在于降低了用户的决策风险，劣势在于可能延长整体交付时间。

决策权分配不仅影响用户体验，也影响了 AI 的责任边界。Claude Code 通过让用户选择，将结果的责任部分转移给了用户；Gemini 通过直接交付，承担了更多的结果责任；ChatGPT 通过试跑机制，在责任分担上采取了折中方案。

### 4. 风险沟通策略：如何管理期望？

**Claude Code：**
> 优点：
> - 免费，可立即执行
> - 反映真实用户搜索意图
>
> 缺点：
> - 无法获取精确搜索量数字
> - 只能推测相对热度
> - 准确性约 60-70%（基于 SERP 特征推断）
>
> 你需要做的：
> - 告诉我是否接受这种‘推断+验证’的方式
> - 后续你可以用 Google Keyword Planner 验证搜索量

**Gemini：**
> 2. 准确性是否可以保证？
> - 趋势准确性：高。这些 FAQ 准确反映了当前美国搜索市场对这些技术概念的最真实困惑点。
> - 量级准确性：中。由于 Google 不会公开具体到每一个问题的精确搜索次数，这里的‘Top 20’是一个基于加权权重的排名。
> - 时效性：极高。针对如 Claude Code 这种 2025-2026 年爆发的新技术，我捕捉的是最新的开发者反馈。

**ChatGPT：**
> ✔ 可以保证的
> * 覆盖90%+真实用户问题
> * 排序大体符合搜索热度
> * 能反映用户真实搜索意图
>
> ❌ 不能保证的
> * 精确排名（第1 vs 第3）
> * 长尾词完整覆盖
> * 实际搜索量绝对值
>
> 甚至业内共识是：
>
> 👉 ‘所有关键词工具本质都是估算’
>
> 👉 ‘Google Search Console 才是真实数据’（但你拿不到全量）

**对比分析：**

三个 AI 在风险沟通上采用了不同的框架和语言策略，影响了用户对结果可靠性的心理预期。

- Claude Code：采用了**优缺点对照式**的风险沟通。它将每种方法的优点和缺点并列展示，并用具体的百分比（60-70%）量化风险。这种方式的优势在于信息对称，用户能基于完整信息做判断；劣势在于可能让用户过度关注缺点，降低对结果的信心。

- Gemini：采用了**分层评级式**的风险沟通。它将准确性拆解为三个维度（趋势、量级、时效性），并用““高/中/极高””这种定性评级来描述。这种方式的优势在于建立信心的同时保持了诚实，劣势在于““高/中/极高””这种定性描述缺乏可验证性。

- ChatGPT：采用了**对比清单式**的风险沟通。它用“✔ 可以保证的”和“❌ 不能保证的”两个清单，清晰划分了能力边界。同时引入了“业内共识”作为风险的背书——通过强调“所有关键词工具本质都是估算”，将风险从个体层面上升到行业层面。这种方式的优势在于降低了用户对“完美结果”的期望，劣势在于可能让用户觉得“既然都不准确，那为什么要用你？”

三个 AI 都没有回避风险，但它们“包装”风险的方式截然不同。Claude Code 用数字量化风险，Gemini 用评级分散风险，ChatGPT 用行业共识稀释风险。

### 5. 数据来源可追溯性：能否验证？

**Claude Code：**
> 方法1： 通过 WebSearch 工具分析 SERP 特征
>
> 我可以搜索每篇文章的核心关键词，然后分析：
> - Google 的 'People Also Ask' (PAA) 框
> - Featured Snippets 中的问题
> - 相关搜索建议
>
> 方法2： 基于我的训练数据推断常见 FAQ
> 根据我的知识库（截止2025年1月），推断每个主题的高频问题。

**Gemini：**
> 我采用了以下'三维交叉验证法'来还原 Top 20 数据：
> - PAA (People Also Ask) 挖掘：实时抓取 Google 搜索结果中由算法自动生成的'用户还问了这些问题'
> - Search Intent Analysis：分析针对这 7 个关键词的'搜索建议（Autocomplete）'
> - 社区热度映射：在开发者和 SEO 专家集中的地方（GitHub， Medium， LinkedIn， Reddit）分析特定话题下的高频互动问题

**ChatGPT：**
> 例如：
> * Ahrefs 的关键词库是基于点击流数据 + 模型推算 ([Ahrefs][1])
> * 工具提供的是月均搜索量（估算值），不是精确值 ([Ahrefs][2])
> * kwrds.ai 就是专门抓PAA问题的 ([kwrds.ai][3])
> * 'Search volume tells you what's popular. Intent and clicks tell you what's worth targeting.' ([Reddit][4])
>

**对比分析：**

三个 AI 在数据来源可追溯性上展现出不同的透明度层级：

- Claude Code：明确说明了使用的工具（WebSearch）和数据来源（训练数据截止2025年1月），并列出了具体的数据点（PAA、Featured Snippets、相关搜索）。但它没有提供具体的搜索结果或链接，用户无法直接验证“AI 是否真的搜索了这些关键词”或“搜索结果是什么”。这种透明度适合技术用户理解方法论，但不足以支持完全的结果验证。

- Gemini：详细描述了三个数据源（PAA、Autocomplete、社区讨论）和具体的平台（GitHub、Medium、LinkedIn、Reddit），但同样没有提供具体的链接或截图。它声称“实时抓取”，但用户无法验证这个“实时”是什么时候，也无法看到原始的抓取数据。

- ChatGPT：是三个 AI 中唯一提供了可点击外部链接的，包括 Ahrefs 的官方文档、kwrds.ai 工具页面、以及 Reddit 的讨论帖。这些链接让用户可以直接验证 ChatGPT 所说的“行业做法”是否属实。但需要注意的是，这些链接证明的是“行业通常怎么做”，而不是“ChatGPT 具体怎么做的”。换句话说，它提供了“行业标准的可追溯性”，但没有提供“自身执行的可追溯性”。

这种可追溯性的缺失是 AI 系统的固有特征。AI 的输出往往是基于大量数据的综合推断，而不是简单的数据检索。要求 AI 为每个输出提供精确的数据溯源，在技术上具有很大挑战。但从用户信任的角度看，提供部分可追溯性（如 ChatGPT 的外部链接）总比完全不提供要好。

### 6. 用户角色假设：为谁服务？

**Claude Code：**
> 推荐方案B： 你提供 API 访问权限
>
> 如果你有以下任一工具的 API：
> - DataForSEO (你的 SEO Skills 文章提到过)
> - Google Search Console API
> - Ahrefs / SEMrush API
>
> 我可以通过 MCP 协议调用这些工具获取精确搜索量数据，准确性可达 90%+。

**Gemini：**
> 建议：如果你是在写文章，可以将这些 FAQ 直接作为文章的 H2 或 H3 标题，或者放在文末的 FAQ Schema 中。这不仅能增加文章的深度，还能极大提高被 Google Featured Snippets (精选摘要) 抓取的概率。

**ChatGPT：**
> 如果你的目标是SEO，而不是'理论Top20'，那更应该追求的是：
> 👉 '覆盖率 + 结构' 而不是 Top20精度
>
> 因为：
> * Google更看内容完整性（topic coverage）
> * FAQ是为了抢：featured snippets、PAA位、long-tail流量

**对比分析：**

三个 AI 在用户角色假设上展现出不同的画像，这反映了它们对“谁会问这个问题”以及“他们真正需要什么”的不同理解。

- Claude Code：假设用户是**技术实践者**。从它提到的 API（DataForSEO、Google Search Console API）、协议（MCP）、以及“你的 SEO Skills 文章”这种表述来看，它认为用户具备一定的技术背景，熟悉 API 集成，并且可能已经在使用专业的 SEO 工具。它提供的解决方案也偏向技术化——通过 API 获取数据，而不是直接给出现成的答案。

- Gemini：假设用户是**内容创作者**。从它的建议“将这些 FAQ 直接作为文章的 H2 或 H3 标题”、“放在文末的 FAQ Schema 中”来看，它认为用户的核心需求是快速将 FAQ 应用到内容创作中。它强调的是“增加文章深度”和“提高被 Featured Snippets 抓取的概率”，这些都是内容 SEO 的典型关注点。确性或方法论的深层关注。

- ChatGPT：假设用户是**SEO 策略师**。从它强调“如果你的目标是SEO，而不是'理论Top20'”、“Google更看内容完整性”、“FAQ是为了抢 featured snippets、PAA位、long-tail流量”来看，它认为用户关注的不是数据本身，而是数据背后的战略价值。它试图引导用户从“追求精确的 Top20”转向“追求覆盖率和结构”，这是一种战略层面的思考。这种假设的优势在于帮助用户理解“为什么”而不只是“是什么”，劣势在于可能让用户觉得“你在改变我的问题”。

这三种用户角色假设都有其合理性，因为“想要获取 FAQ 数据”的用户群体本身就是多元的。技术博主可能确实需要 API 集成，内容营销人员可能确实需要快速应用，SEO 顾问可能确实需要战略指导。问题在于，当 AI 做出错误的用户角色假设时（比如对非技术用户谈论 MCP 协议），就会造成沟通障碍。

理想的 AI 交互应该在初期就明确用户角色，或者提供多种路径供不同角色的用户选择。但在实际对话中，AI 往往需要基于有限的信息快速做出假设，这就导致了我们看到的这种差异。

## 不同AI的交付结果

在分析完三个 AI 的处理逻辑后，让我们来看看它们最终交付的结果有什么差异。这些差异不仅体现在数据的呈现形式上，更体现在它们对“什么是有价值的交付物”的不同理解。

### 1. 交付形式对比：结构化 vs 叙事化

**Claude Code：**
- 提供了结构化的表格，包含问题、置信度标注(🔥/🔸/🔹)、搜索意图分类
- 每个问题都有元数据标注，便于后续筛选和优先级排序
- 在交付前提供了详细的方法论说明和风险提示

**Gemini：**
- 在每个主题前提供了一句话总结(如“CLI vs GUI vs Skills： 开发者在选择交互方式时的核心困惑”)
- 问题以列表形式呈现，没有额外的元数据
- 将方法论说明放在文末，优先展示结果

**ChatGPT：**
- 提供了纯问题列表，没有置信度或意图分类
- 没有提供一句话总结或其他辅助信息
- 在交付前进行了大量的行业背景铺垫

**对比分析：**

三个 AI 在交付形式上的差异反映了它们对“用户需要什么”的不同假设。

- Claude Code：将每个 FAQ 视为一个数据点，并为其附加了可操作的元数据(置信度、搜索意图)。这种结构化的设计让用户可以根据自己的需求进行二次筛选——比如只选择高置信度的问题，或者只关注“How-to”类型的问题。这种方式的优势在于灵活性和可扩展性，劣势在于增加了信息密度，可能让用户感到overwhelmed。

- Gemini：在每个主题前提供的一句话总结(如“开发者在选择交互方式时的核心困惑”)帮助用户快速理解这20个问题的共同主题。这种设计体现了对“内容创作者”用户角色的理解——他们需要的不只是问题列表，还需要对问题背后的用户意图有整体把握。这种方式的优势在于提供了更高层次的洞察，劣势在于这种总结本身也是主观的。

- ChatGPT：只提供问题列表，没有任何额外的标注或总结。这种设计可能基于两种考虑：一是避免过度解读(让用户自己判断)，二是将重点放在问题本身的质量上。这种方式的优势在于清晰简洁，劣势在于缺少了可操作的指导信息。

从用户体验的角度看，这三种交付形式服务于不同的使用场景：如果你需要进行数据分析或优先级排序，Claude Code 的结构化数据更有价值;如果你需要理解用户的整体需求趋势，Gemini 的主题总结更有帮助;如果你只是需要一个问题清单来启发思考，ChatGPT 的极简列表就足够了。

### 2. 数据量与完整性

三个 AI 都按要求为7个主题各提供了20个问题，总计420个问题。通过对比分析，我发现：

**问题重复率：**
- 三个 AI 之间的问题重复率很低(约10-15%)，说明它们确实采用了不同的数据源或推断逻辑
- 重复的问题主要集中在最基础的概念性问题上(如“What is X?”“How does X work?”)

**问题覆盖度：**
- Claude Code 的问题更偏向技术实现细节(如“How to debug Skills?”“What APIs are available?”)
- Gemini 的问题更平衡，既包含概念性问题也包含实践性问题
- ChatGPT 的问题更偏向用户决策场景(如“When to use X vs Y?”“Is X suitable for beginners?”)

**数据完整性：**
- 所有 AI 都完成了140个问题的交付，没有出现数据缺失
- Claude Code 额外提供了置信度和搜索意图分类，数据维度更丰富
- Gemini 额外提供了主题总结，增加了语义层面的完整性

从数据质量的角度看，三个 AI 都达到了基本的交付标准(数量、相关性)，但在数据的“附加价值”上有明显差异。Claude Code 提供了更多可操作的元数据，Gemini 提供了更多语义层面的洞察，ChatGPT 则保持了数据的纯粹性。
