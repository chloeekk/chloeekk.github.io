How to Exclude Internal Traffic in Google Analytics 4 by IP address

Implementing Google Analytics 4 also tracks user interactions, including those of the implementing team, which is the standard setup. This becomes particularly challenging for larger organizations with more employees, as each internal interaction can distort the analytics data. Fortunately, Google Analytics 4 offers a solution to filter out this internal traffic.

However, this process has its complexities. For instance, if you exclude your own interactions, they won’t appear in DebugView when testing with Google Tag Manager (GTM). While the GA4 documentation suggests that using a developer filter should resolve this issue, practical experiences may vary.

Moreover, when excluding multiple team members, relying solely on IP addresses may not be the most effective approach. This article aims to guide you through the steps to effectively exclude internal traffic in Google Analytics 4, addressing these specific challenges. Additionally, the article will discuss the IP address method as an alternative solution.

How Internal Traffic Filter works

When internal traffic is specified in a Data Stream’s tagging settings, the Google Analytics 4 tracking code on your website begins appending the parameter ‘tt=internal’ to each request, signifying ‘traffic type’ or ‘traffic_type’.

In the default settings, found under Admin > Data Settings > Data Filters, the Internal Traffic Filter is preset to omit all hits that carry the ‘traffic_type=internal’ parameter, also known as ‘tt=internal’. This exclusion process is activated once you set the filter to “Active”.

Beginning with the traditional approach, this method is effective for businesses with a smaller team, allowing for easy exclusion of their hits using IP addresses. However, this approach becomes less practical when dealing with a large number of varying IP addresses or when some of the IP addresses are dynamic.

Step 1. Define your internal IP addresses

Navigate to Admin > Data Streams in Google Analytics 4 and choose your website’s data stream. Then click Configure tag settings.

Next, select “Show all” and choose the “Define internal traffic” option, where you can establish new rules for internal traffic.

Step 2. Create a developer filter in GA4

You may be curious about how a developer filter differs from an internal traffic filter in GA4, as both exclude data from GA4’s standard reports by evaluating different parameters.

The developer filter assesses the debug_mode or debug_event parameters, whereas the internal traffic filter focuses on the traffic_type parameter.

However, the main distinction lies in the visibility of data: developer traffic is visible in GA4’s DebugView, but internal traffic isn’t displayed in any section.

In Google Analytics 4, navigate to Admin > Data Settings > Data Filters > Create Filter. Choose ‘Developer Traffic’ and input the following settings. You can use any name you prefer in the “Data Filter Name” field. Set this filter to ‘Testing’ mode and then save your changes.

Step 3: Set up a Lookup Table in GTM to Modify the ‘traffic_type’ Parameter

Google Analytics 4 automatically appends a ‘tt=internal’ (traffic_type) parameter to all hits originating from any of your internal IP addresses.

We will create a Lookup Table variable in GTM to adjust the value of the parameter. If the GTM preview mode is active, we will alter the value of ‘tt’ to ‘developer’ (or another chosen term). However, if Debug Mode is off, we won’t make any changes, allowing GA4 to manage it.

Start by creating an Undefined variable. In GTM, navigate to Variables > New > Undefined, and then save this variable, which you can simply name ‘Undefined’.

Next, proceed to Variables > Configure (located in the Built-in Variables section) in Google Tag Manager and activate the Debug Mode variable.

Lastly, navigate to Variables > New > Lookup Table in Google Tag Manager and input the specified settings:

When the GTM preview mode is active, this variable will yield a true value, leading the lookup table variable to return the term “developer”. If the GTM Preview mode is not in use, the Debug Mode variable will result in false, and hence, the return value will be undefined. This is the reason for creating the Undefined variable.

Concluding the process, navigate to your Google tag and add the following parameter: ‘traffic_type’. The value for this should be the lookup table you’ve recently set up.

Step 4. Test the filters

Now it’s time to test the setup. Activate the preview mode in GTM and visit your website, ensuring that you are accessing it from an IP address specified in the Internal Traffic section of your Data Stream. Next, open the DebugView in GA4.

You should be able to observe hits originating from your device. Select any event in the event stream and review the ‘traffic_type’ parameter. Its value should display as ‘developer’.

Additionally, within the same event, the Debug Mode parameter should also be visible.

Step 5. Activate both filters

The final step involves activating both filters. In Google Analytics, navigate to Admin > Data Settings > Data Filters. Here, set both the Internal Traffic and Developer Traffic filters to active. This can be done by opening each filter, modifying its Filter State, and then saving the changes. Alternatively, you can simply activate them directly from the list.
